# PostgreSQL 双机热备项目已知风险

> 适用范围：当前 PostgreSQL HA 双机热备方案，基于 PostgreSQL + repmgr + Keepalived + Docker 实现。
> 说明：本文用于研发自测、交付评审、上线前风险确认和运维交接。

## 1. 大库全量 Clone 期间 WAL 被回收风险

主库 A 持续大量写入，备库 B 全量 clone 时间很长时，clone 完成后 B 需要继续追赶的 WAL 可能已经被 A 回收，导致 B 无法进入 streaming，进而再次触发全量 clone。

## 2. 默认 WAL 保留能力不足

当前默认 `wal_keep_size=512MB`，对小数据量够用，但对大库、高写入、长时间 clone 场景明显不够，需要按业务写入速度重新评估。

## 3. WAL Archive 默认未真正启用

当前默认归档命令等价于不保存真实 WAL。如果主库回收了 WAL，备库没有归档可追，恢复能力受限。

## 4. Replication Slot 未默认启用

没有 replication slot 时，主库不会长期为备库保留 WAL。备库离线时间过长或 clone 太慢时，可能追不上主库。

## 5. 启用 Replication Slot 后存在磁盘打满风险

如果后续开启 slot，备库长时间不可用时，主库会持续保留 WAL，可能导致主库磁盘被 WAL 打满，需要配套监控。

## 6. 异步复制存在最后事务丢失风险

当前方案属于常见的一主一备异步复制。主库突然断电时，尚未复制到备库的最新事务可能丢失，不能保证 `RPO=0`。

## 7. 无第三方仲裁，网络分区存在脑裂风险

当前是双节点架构，没有仲裁节点。如果两台机器之间网络分区，极端情况下可能出现双方对主备状态判断不一致的风险。

## 8. VIP 漂移依赖宿主机网络环境

VIP 是否能正常漂移，依赖同网段网络、ARP、Keepalived、容器 `NET_ADMIN` 权限、宿主机防火墙等环境。数据库升主成功不代表 VIP 一定能被业务访问到。

## 9. 切换不是零中断

repmgr 和 Keepalived 都有检测时间，主库故障后需要一段时间完成检测、promote、VIP 漂移和业务重连。业务侧必须有连接池重试机制。

## 10. 全量 Clone 时间不可控

数据量越大、主库写入越高、网络越慢，clone 时间越长。clone 期间备库不可用，恢复窗口会变大。

## 11. 全量 Clone 可能反复重试

当前实现已经保护了原 `current` 数据目录，clone 失败不会直接清空原数据。但如果根因是 WAL 不足、网络不稳或主库压力过高，可能出现反复 clone、反复失败。

## 12. 旧主回归可能降级为全量 Clone

旧主恢复时会优先走 `pg_rewind / repmgr node rejoin`，但如果 WAL 缺失、时间线差异过大或数据目录状态异常，会降级为全量 clone，恢复时间会变长。

## 13. 只保留最近一份旧数据，不等于备份

当前 clone 成功后只保留 `latest` 一份旧数据，下一次 clone 成功会替换它。它只能作为短期回退参考，不能替代正式备份。

## 14. 磁盘空间风险

数据目录、clone 临时目录、`latest` 保留目录、WAL、PostgreSQL 日志、Docker 日志都会占用磁盘。大库场景必须监控数据卷和宿主机磁盘。

## 15. 大库启动仍可能被权限修复拖慢

当前已经减少了递归 `chown` 的触发范围，但如果检测到 `PGDATA` owner 异常，仍会递归修复权限。大库场景下这会影响启动和恢复耗时。

## 16. 企业微信通知不能作为唯一判断依据

企业微信可能因为网络、Webhook、限流或配置问题发送失败。故障判断仍应以 PostgreSQL 角色、VIP、repmgr 状态和日志为准。

## 17. 日志增强不等于自动修复

当前日志能看到 clone、rejoin、VIP、Keepalived、企业微信等过程，但日志只是辅助排障，不能替代监控告警和人工应急流程。

## 18. Docker Host Network 和权限依赖较强

当前方案依赖 `network_mode: host`、`NET_ADMIN`、Keepalived。部分容器平台、安全策略或云环境可能不允许这样运行。

## 19. 安全配置偏内网信任模式

当前 `pg_hba` / repmgr 配置更偏向内网高可用部署便利性。如果部署环境不是可信内网，需要加强密码、访问控制和网络隔离。

## 20. 双机热备不能解决误删误改

如果业务误删除、误更新数据，这类逻辑错误会复制到备库。必须单独做定期备份、PITR 和恢复演练。

## 21. 无法自动合并双主写入冲突

当前方案是单主架构，不是多主数据库。如果发生异常双主并且两边都有写入，系统不能自动合并冲突数据。

## 22. 测试覆盖不能等同于生产全覆盖

当前已经做过多轮切换、大写入、clone 失败保护、VIP 跟随、企业微信推送测试，但生产环境的真实网络、磁盘、负载和故障形态仍可能不同，需要上线前做现场压测和演练。

# 核心风险总结

当前项目最需要重点关注两类风险：

## 1. 大库高写入场景下的 WAL 保留风险

如果全量 clone 时间较长，而主库持续产生大量 WAL，备库 clone 完成后可能无法追上主库，导致再次全量 clone。生产环境需要重点评估：

- `wal_keep_size`
- WAL archive
- replication slot
- 磁盘空间监控
- clone 后 `wal_receiver=streaming` 校验

## 2. 双节点架构下的网络分区风险

当前是双节点架构，没有第三方仲裁。极端网络分区场景下，需要明确人工处置流程，避免出现双主或错误切换。

# 建议优先改进项

## P1：生产上线前必须确认

- 增大 `wal_keep_size` 或启用可靠 WAL archive。
- 明确是否启用 replication slot，并增加磁盘监控。
- 对数据卷、WAL、Docker 日志配置磁盘告警。
- 业务侧配置数据库连接重试和超时。
- 现场验证 VIP 漂移和 ARP 更新是否正常。
- 建立定期备份和恢复演练流程。

## P2：建议后续优化

- clone 成功后强制校验 `wal_receiver=streaming`。
- 对 WAL 缺失导致的重复 clone 增加更明确日志。
- 优化大库场景下 `chown -R` 权限修复策略。
- 增加网络分区场景下的人工确认或仲裁机制。
- 增强安装文档中的生产风险提示。

## P3：长期能力建设

- 接入 Prometheus / 日志平台 / 企业微信告警闭环。
- 增加自动化巡检脚本。
- 增加定期 HA 演练流程。
- 评估是否需要三节点架构或外部仲裁。
